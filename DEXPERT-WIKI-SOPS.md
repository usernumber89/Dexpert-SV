## Misión, Visión y Valores

### Misión

Conectar el talento joven de El Salvador con pequeñas y medianas empresas, brindando una plataforma donde estudiantes adquieran experiencia profesional real y las PYMEs encuentren el talento que necesitan para crecer.

### Visión

Ser el puente principal entre la academia y el mundo laboral en Centroamérica, donde ningún talento joven se quede sin oportunidad por falta de experiencia y ninguna PYME se quede sin crecer por falta de talento.

### Valores

| Valor | Descripción |
|-------|-------------|
| **Talento sin etiquetas** | Creemos en el potencial, no en los títulos. Un joven sin experiencia puede tener el talento que una empresa necesita. |
| **Conexiones reales** | No somos una plataforma de cursos. Somos un puente entre personas que se necesitan mutuamente. |
| **Crecimiento mutuo** | Cuando un estudiante aprende y una PYME crece, ganamos todos. El éxito es compartido o no es éxito. |
| **Hecho en El Salvador** | Nuestro corazón late al ritmo de nuestro país. Cada funcionalidad, cada decisión, cada detalle está pensado para el ecosistema salvadoreño. |
| **Innovación con propósito** | Usamos IA y tecnología de punta, pero siempre al servicio de conectar personas y generar oportunidades reales. |

---

## Cultura Organizacional

- **Comunicación directa y clara**: Preferimos decir las cosas como son, con respeto y sin rodeos.
- **Autonomía responsable**: Cada miembro del equipo tiene la libertad de decidir cómo hacer su trabajo, pero asume la responsabilidad por los resultados.
- **Mejora continua**: Todo puede mejorarse. Buscamos activamente feedback para evolucionar la plataforma y los procesos.
- **Obsesión por el usuario**: Cada decisión se toma pensando en cómo impacta al estudiante y a la PYME.
- **Documentación primero**: Si no está documentado, no existe. Todo proceso debe tener su SOP o documento de referencia.

## Metodologías de Trabajo

### Desarrollo

- **Git Flow simplificado**: `main` (producción) → `develop` (integración) → `feature/*` (ramas por funcionalidad)
- **Conventional Commits**: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`
- **Code Review**: Todo PR debe tener al menos 1 approval antes de mergear
- **Pre-commit hooks**: Linting (ESLint) y type checking (TypeScript) antes de commitear
- **Despliegues**: Automáticos a producción desde `main` vía Vercel

### Ciclo de desarrollo

1. **Planificación**: Se define la tarea/feature en GitHub Issues
2. **Diseño**: Si aplica, mockups/flow en Figma
3. **Implementación**: Rama `feature/xxx`, desarrollo con TDD cuando aplique
4. **Review**: PR a `develop`, code review, pruebas
5. **QA**: Pruebas en ambiente de staging
6. **Release**: Merge a `main`, despliegue automático a producción
7. **Monitoreo**: Revisar logs, métricas y feedback post-despliegue

### Reuniones

| Reunión | Frecuencia | Duración | Participantes |
|---------|-----------|----------|---------------|
| **Daily Standup** | Diaria | 10 min | Equipo técnico |
| **Sprint Planning** | Quincenal | 30 min | Todo el equipo |
| **Sprint Review** | Quincenal | 20 min | Todo el equipo |
| **Retrospectiva** | Quincenal | 30 min | Todo el equipo |

## Políticas Internas

### Seguridad

- **Nunca compartir secrets**: API keys, tokens, passwords — todo debe ir en variables de entorno.
- **2FA obligatorio**: En Supabase, GitHub, Vercel y Wompi.
- **Acceso por rol**: Solo el CTO y desarrolladores senior tienen acceso a producción.
- **Rotación de keys**: Las API keys de servicios externos se rotan cada 90 días.
- **Logs de auditoría**: Todas las acciones administrativas quedan registradas en `audit_logs`.

### Código

- **No hacer push directo a `main`** — siempre usar PRs.
- **No subir `.env` ni secrets al repositorio**. El `.gitignore` ya los excluye.
- **Documentar cambios significativos** en el PR description.
- **Mantener cobertura de tipos**: No usar `any` en TypeScript salvo excepciones justificadas.

### Datos de Usuarios

- **Privacidad**: No compartir datos de usuarios fuera del equipo. Cumplir con leyes de protección de datos de El Salvador.
- **Retención**: Los datos se conservan mientras la cuenta esté activa. Al solicitar baja, se anonimizan a los 30 días.
- **Soporte**: Solo el equipo de soporte tiene acceso a datos personales de usuarios, y únicamente para resolver incidencias.

## Arquitectura del Sistema

```
Usuario (Browser)
      │
      ▼
  Next.js 16 (App Router)
  ┌──────────────────────────────────────────┐
  │  Server Components / Client Components   │
  │  Server Actions / API Routes             │
  └──────┬─────────────────────────┬─────────┘
         │                         │
         ▼                         ▼
  ┌────────────┐          ┌──────────────┐
  │  Supabase   │          │   Servicios   │
  │  Auth + DB  │          │  Externos     │
  │  + Storage  │          │              │
  │  + Realtime │          │  • Groq AI   │
  └────────────┘          │  • Wompi     │
                          │  • Stripe    │
                          │  • UploadT.  │
                          │  • Svix      │
                          └──────────────┘
```

### Flujo de datos principal

1. El usuario interactúa con la UI (Server Components o Client Components)
2. Las acciones se ejecutan via **Server Actions** (mutaciones) o **API Routes** (lecturas/complejas)
3. Los datos se persisten en **Supabase (PostgreSQL)**
4. Las integraciones externas se llaman desde **API Routes** del servidor (nunca desde el cliente)
5. Los webhooks de Wompi llegan a `/api/wompi/webhook` y actualizan el estado

## Flujo de Datos Clave

### Ciclo de vida de un Proyecto

```
PYME crea proyecto → (opcional: IA genera brief) → Publica → Estudiantes aplican
→ PYME acepta estudiante(s) → Proyecto "En Progreso"
→ PYME crea milestones por estudiante
→ Estudiante sube entregables → PYME revisa y aprueba/rechaza
→ Todos los milestones aprobados → Proyecto "Completado"
→ Certificado generado automáticamente
→ Entrada de portafolio creada automáticamente
→ XP otorgada al estudiante (150 XP)
```

### Ciclo de Pago

```
Usuario selecciona plan/producto → /api/wompi/checkout genera enlace Wompi
→ Usuario es redirigido a Wompi → Paga → Wompi envía webhook a /api/wompi/webhook
→ Webhook valida HMAC → Procesa la compra → Actualiza BD → Redirige a success
```

### Ciclo de Simulación con IA

```
Estudiante selecciona escenario → IA genera mensaje de bienvenida del "cliente virtual"
→ Estudiante conversa con el cliente IA
→ (opcional) IA genera change requests dinámicos
→ Estudiante sube entregable → IA evalúa en 5 criterios
→ Resultados: score, rúbrica, fortalezas, mejoras
→ Entrada de portafolio y experiencia actualizadas automáticamente
```

## SOP-001: Onboarding de Nuevo Colaborador

### Objetivo

Integrar a un nuevo miembro del equipo a la operación de Dexpert en el menor tiempo posible, asegurando que comprenda la cultura, las herramientas y sus responsabilidades.

### Responsable

- **Líder de equipo** (CEO, CTO según el rol)
- **Recursos Humanos** (si aplica)

### Herramientas Necesarias

- Correo corporativo
- Acceso a GitHub
- Acceso a Supabase (si aplica)
- Acceso a Vercel (si aplica)
- Invitación a Notion
- Invitación a Slack/Discord

### Pasos

1. **Preparación previa al inicio (3 días antes)**
   - El líder de equipo crea los accesos necesarios (GitHub, Supabase, Vercel, Notion, Slack).
   - Prepara la estación de trabajo: laptop, monitores, accesorios.
   - Asigna un "buddy" (compañero de onboarding) que guiará al nuevo colaborador en la primera semana.

2. **Día 1 — Inducción general**
   - Presentación del equipo (15 min, cada miembro se presenta brevemente).
   - Revisión de Misión, Visión y Valores (leer esta Wiki completa).
   - Tour por las herramientas: Slack, Notion, GitHub, ambiente de desarrollo.
   - El buddy comparte el link a esta Wiki y a los SOPs relevantes para su rol.
   - Configuración del ambiente local de desarrollo (si aplica):
     - `git clone` del repositorio
     - `npm install`
     - Copiar `.env.example` a `.env.local` y completar variables
     - `npm run dev` para verificar que corre localmente

3. **Día 2-3 — Inmersión en producto**
   - El nuevo colaborador crea una cuenta de prueba como Estudiante y como PYME.
   - Realiza una simulación completa con IA.
   - Crea un proyecto como PYME, aplica como estudiante, completa el flujo.
   - Explora el panel de administración (si aplica a su rol).
   - Lee los SOPs relevantes a su área.

4. **Día 4-5 — Primera tarea**
   - Se asigna una tarea pequeña y real (bug fix o feature pequeña).
   - El buddy guía el proceso: rama → desarrollo → PR → review → merge.
   - El líder de equipo da feedback sobre el resultado.

5. **Semana 2 — Autonomía progresiva**
   - Se asignan tareas de complejidad creciente.
   - Se espera que el colaborador ya pueda trabajar de forma independiente.
   - Reunión de feedback con el líder a los 15 días.

### Métricas de Éxito

- El colaborador puede ejecutar el ambiente local en el Día 1.
- Completó al menos 1 tarea real en la primera semana.
- Reporta sentirse "cómodo" con las herramientas y procesos en la reunión de 15 días.
- No necesita preguntar sobre procesos documentados en SOPs después de la Semana 2.

### Notas Importantes

- Si el colaborador es técnico (desarrollador), el buddy debe ser otro desarrollador senior.
- Si el colaborador es no técnico (soporte, marketing), enfocar el onboarding en la plataforma desde el rol del usuario final.
- Documentar cualquier mejora sugerida durante el onboarding y actualizar este SOP.

## SOP-002: Atención al Cliente — Reembolsos

### Objetivo

Procesar solicitudes de reembolso de manera consistente, justa y documentada, minimizando la fricción con el usuario y protegiendo los ingresos de la plataforma.

### Responsable

- **Community Manager / Soporte**
- **CEO** (para aprobaciones de montos mayores)

### Herramientas Necesarias

- Panel de administración en Dexpert (sección Finanzas > Transacciones)
- Dashboard de Wompi (`https://comercios.wompi.sv`)
- Correo electrónico corporativo

### Pasos

1. **Recepción de solicitud**
   - El usuario solicita reembolso vía email, chat o formulario de contacto.
   - Registrar: nombre, email, monto, producto/servicio, motivo, fecha de compra.
   - Clasificar el motivo:
     - **Error de plataforma**: cargo duplicado, servicio no entregado.
     - **Insatisfacción**: el producto no cumplió expectativas.
     - **Cambio de opinión**: el usuario ya no necesita el servicio.
     - **Fraude**: cargo no reconocido.

2. **Verificación elegibilidad**
   - Verificar en el panel de administración la transacción original (tabla `purchases` o `credit_purchases`).
   - Verificar en Wompi el estado de la transacción.
   - Aplicar política de reembolso:
     - **Créditos PYME no utilizados**: reembolso completo si no se ha publicado ningún proyecto con esos créditos.
     - **Créditos PYME usados parcialmente**: reembolso proporcional (créditos no usados / total * monto).
     - **Student Boost**: reembolso completo si se solicita dentro de las primeras 48 horas.
     - **Certificados**: No reembolsables una vez emitidos y verificados.
     - **Portafolio PDF**: No reembolsable una vez descargado.

3. **Resolución (2 escenarios)**

   **Escenario A: Procede el reembolso**
   - Iniciar sesión en Wompi Dashboard.
   - Ir a Transacciones → Buscar la transacción → "Reembolsar".
   - Ingresar monto a reembolsar y motivo.
   - Confirmar reembolso.
   - Actualizar la BD manualmente si es necesario (marcar registros como reembolsados).
   - Notificar al usuario por correo: "Hemos procesado tu reembolso por $X. El dinero se verá reflejado en tu método de pago en 3-5 días hábiles."
   - Registrar en el log interno (fecha, monto, usuario, motivo, agente que procesó).

   **Escenario B: No procede el reembolso**
   - Responder al usuario explicando la política aplicable y el motivo.
   - Ofrecer alternativas: crédito en la plataforma, descuento en próxima compra, o ayuda adicional con el producto.
   - Si el usuario insiste, escalar a CEO.

4. **Documentación**
   - Registrar el caso en el sistema de tickets (si aplica).
   - Si se otorgó reembolso, actualizar el registro de la transacción en la BD con estado `refunded`.

### Métricas de Éxito

- Tiempo de respuesta inicial < 4 horas hábiles.
- Reembolso procesado en < 24 horas desde la aprobación.
- Tasa de reembolsos < 5% de las transacciones mensuales.
- Usuario notificado en cada etapa del proceso.

### Notas Importantes

- Siempre verificar que no sea un intento de fraude (múltiples solicitudes del mismo usuario, tarjetas robadas).
- Los reembolsos en Wompi pueden tardar hasta 10 días hábiles en reflejarse. Informar al usuario.
- No procesar reembolsos sin antes consultar al CEO si el monto excede los $50.

## SOP-003: Atención al Cliente — Quejas y Escalación

### Objetivo

Gestionar quejas de usuarios (estudiantes y PYMEs) de forma estructurada, asegurando que cada queja sea atendida, documentada y resuelta o escalada según su criticidad.

### Responsable

- **Community Manager / Soporte** (primera línea)
- **CEO / CTO** (escalación)

### Herramientas Necesarias

- Panel de administración de Dexpert
- Correo electrónico
- Slack (canal #soporte-escalaciones)

### Pasos

1. **Recepción y clasificación**
   - Recibir la queja por el canal que sea (email, formulario, redes sociales).
   - Clasificar la queja en una de estas categorías:
     - **Técnica**: error en plataforma, bug, funcionalidad no funciona.
     - **Entre usuarios**: conflicto entre estudiante y PYME (entregables no aceptados, comunicación inapropiada, impagos).
     - **Contenido**: proyecto inapropiado, spam, perfil falso.
     - **Facturación**: cargo no reconocido, error en precio, problema con créditos.

2. **Respuesta inicial (máximo 2 horas hábiles)**
   - Acusar recibo: "Hemos recibido tu queja y la estamos revisando. Te daremos respuesta en máximo 24 horas hábiles."
   - Asignar un ticket o ID interno para seguimiento.

3. **Investigación**
   - Revisar los registros relevantes en la BD (proyecto, aplicación, mensajes, transacciones).
   - Si es técnica, involucrar al CTO/desarrollador.
   - Si es entre usuarios, revisar el historial de comunicación y milestones.
   - Documentar hallazgos.

4. **Resolución**
   - Determinar la acción correctiva:
     - **Técnica**: corregir bug, restaurar datos, compensar con créditos si hubo pérdida.
     - **Entre usuarios**: mediar, cancelar proyecto si es necesario, banear si aplica.
     - **Contenido**: eliminar contenido, suspender cuenta temporal/permanentemente.
     - **Facturación**: reembolsar, corregir, otorgar créditos.
   - Comunicar la resolución al usuario.
   - Confirmar con el usuario que la solución es satisfactoria.

5. **Escalación**
   - Escalar a CEO si:
     - La queja implica riesgo legal.
     - El monto en disputa excede los $100.
     - El usuario amenaza con acciones legales o exposición pública negativa.
     - No se llega a una resolución en 5 días hábiles.

6. **Cierre**
   - Una vez resuelto y confirmado, cerrar el caso.
   - Registrar en el log de quejas (fecha, usuario, categoría, resolución, agente).
   - Si la queja revela un problema sistémico, crear un issue en GitHub para la mejora.

### Métricas de Éxito

- Tiempo de primera respuesta < 2 horas hábiles.
- Tiempo de resolución < 48 horas para quejas no escaladas.
- 90% de usuarios satisfechos con la resolución (encuesta post-cierre).
- Cero quejas sin documentar.

### Notas Importantes

- Mantener siempre un tono profesional y empático. El usuario frustrado no es el enemigo.
- Registrar todo: fechas, decisiones, acuerdos. Esto protege al usuario y a la empresa.
- Si un mismo usuario presenta 3+ quejas en 30 días, escalar automáticamente a CEO para revisión de cuenta.

## SOP-004: Moderación de Contenido en Proyectos

### Objetivo

Mantener la calidad y seguridad del contenido publicado en la plataforma (proyectos, perfiles, mensajes), eliminando spam, lenguaje ofensivo y contenido inapropiado.

### Responsable

- **Admin** (a través del panel de administración)
- **Sistema automático** (detección de profanidad y spam)

### Herramientas Necesarias

- Panel de administración > Proyectos
- Sistema de moderación automática (`lib/admin/profanity.ts`)
- Slack (alertas de contenido detectado)

### Pasos

1. **Detección automática (pre-publicación)**
   - Al crear o editar un proyecto, el sistema analiza título y descripción:
     - **Detección de profanidad**: español e inglés (lista de palabras prohibidas y variantes).
     - **Detección de spam**: URLs excesivas, @menciones, keywords sospechosas.
   - Si se detecta contenido prohibido:
     - El proyecto se marca como `flagged` y no se publica.
     - Se notifica al usuario: "Tu proyecto contiene lenguaje que no permite la plataforma. Revísalo e inténtalo de nuevo."
     - Se envía alerta al admin via notificación en el panel.

2. **Revisión manual (post-publicación)**
   - El admin revisa periódicamente la sección de Proyectos en el panel.
   - Busca proyectos marcados como `flagged` o reportados por usuarios.
   - También revisa proyectos activos aleatoriamente (10% semanal).

3. **Acción sobre contenido inapropiado**
   - **Leve** (una palabra inapropiada, spam leve): Editar el contenido, eliminar la parte problemática, notificar al usuario.
   - **Grave** (discurso de odio, contenido ilegal, acoso): Eliminar el proyecto inmediatamente. Suspender la cuenta del usuario. Escalar a CEO.
   - **Spam recurrente**: Si el mismo usuario publica repetidamente contenido spam, suspender la cuenta.

4. **Apelación**
   - El usuario puede apelar la decisión respondiendo al correo de notificación.
   - El admin revisa la apelación en < 24 horas.
   - Si la apelación procede, se restaura el proyecto. Si no, se confirma la decisión.

### Métricas de Éxito

- Contenido inapropiado eliminado en < 2 horas desde la detección.
- 0 proyectos con contenido inapropiado visibles por más de 24 horas.
- Tasa de reincidencia < 5% entre usuarios notificados.

### Notas Importantes

- El sistema automático puede tener falsos positivos. Revisar manualmente cualquier flag antes de tomar acción irreversible.
- No revelar públicamente la lista completa de palabras prohibidas para evitar evasión.
- En caso de contenido ilegal (amenazas, incitación a la violencia, contenido sexual), documentar y contactar a las autoridades correspondientes.

## SOP-008: Gestión de Incidencias Técnicas

### Objetivo

Responder y resolver incidencias técnicas en la plataforma (bugs, caídas, errores) de forma estructurada, minimizando el impacto en los usuarios.

### Responsable

- **CTO**
- **Desarrolladores**

### Herramientas Necesarias

- GitHub Issues (tracking)
- Vercel Dashboard (logs, deploys)
- Supabase Dashboard (base de datos, errores)
- Slack (canal #incidencias)

### Pasos

1. **Detección**
   - La incidencia puede ser detectada por:
     - Reporte de usuario (soporte lo recibe y crea ticket).
     - Monitoreo automático (alertas configuradas en el panel admin).
     - El equipo durante uso normal.
   - Clasificar por severidad:
     - **Crítica (P0)**: Plataforma caída, usuarios no pueden iniciar sesión, pagos no procesan.
     - **Alta (P1)**: Funcionalidad principal no funciona (simulaciones, proyectos).
     - **Media (P2)**: Funcionalidad secundaria con fallo (notificaciones, perfil).
     - **Baja (P3)**: Bug cosmético, error menor sin impacto funcional.

2. **Respuesta inicial**
   - **P0**: Responder en < 15 minutos. Slack: @here en #incidencias. CTO lidera.
   - **P1**: Responder en < 1 hora. Slack en #incidencias.
   - **P2**: Responder en < 4 horas. Crear GitHub Issue.
   - **P3**: Responder en < 24 horas. Crear GitHub Issue.

3. **Investigación y resolución**
   - Revisar logs en Vercel (Deployments > Functions > Logs).
   - Revisar errores en Supabase (Database > Query Performance, Auth > Logs).
   - Identificar la causa raíz.
   - Implementar fix en rama `fix/xxx`.
   - PR a `develop` → revisión → merge a `main`.
   - Despliegue a producción.

4. **Comunicación**
   - **P0/P1**: Comunicar a usuarios afectados via correo/notificación en plataforma: "Estamos experimentando un problema técnico. Nuestro equipo ya está trabajando en resolverlo."
   - **P2/P3**: Comunicar al usuario que reportó: "Hemos registrado el bug y lo resolveremos en la próxima actualización."

5. **Post-mortem (solo P0)**
   - Dentro de las 48 horas posteriores a la resolución:
     - Documentar: cronología, causa raíz, impacto, solución, lecciones aprendidas.
     - Compartir con el equipo.
     - Actualizar procesos/monitoreo para prevenir recurrencia.

### Métricas de Éxito

- P0 resuelto en < 2 horas.
- P1 resuelto en < 8 horas.
- P2 resuelto en < 48 horas.
- 100% de P0 con post-mortem documentado.

### Notas Importantes

- Nunca hacer fix directo en `main` sin PR (salvo P0 crítico, y aun así crear el PR inmediatamente después).
- Si un fix requiere migración de BD, ejecutarla en Supabase Dashboard con la consulta exacta y verificada.
- Documentar workarounds temporales para que soporte pueda ayudar a usuarios mientras se despliega el fix definitivo.

## SOP-010: Backup y Recuperación de Datos

### Objetivo

Garantizar la integridad y disponibilidad de los datos de la plataforma mediante backups regulares y un plan de recuperación documentado.

### Responsable

- **CTO**

### Herramientas Necesarias

- Supabase Dashboard (Database > Backups)
- Almacenamiento externo (para backups adicionales)
- Scripts de backup en `scripts/`

### Pasos

1. **Backups automáticos (Supabase)**
   - Supabase realiza backups automáticos diarios de PostgreSQL (retención: 7 días para el plan estándar).
   - Verificar semanalmente que los backups se están generando:
     - Supabase Dashboard → Database → Backups.
     - Debe haber un backup exitoso por cada día.

2. **Backups manuales (semanales)**
   - Cada domingo a las 23:00, ejecutar backup manual:
     ```
     pg_dump --dbname=postgresql://xxxx:xxxx@xxxx.supabase.co:5432/postgres > backup-dexpert-YYYY-MM-DD.sql
     ```
   - Subir el archivo a almacenamiento externo seguro (Google Drive corporativo o S3).
   - Retención: mantener backups semanales de los últimos 3 meses, mensuales de los últimos 12 meses.

3. **Backup de archivos (Supabase Storage)**
   - Los archivos subidos (avatares, logos) están en Supabase Storage.
   - Supabase Storage se respalda automáticamente junto con la BD.
   - Verificar mensualmente que los archivos críticos están accesibles.

4. **Plan de recuperación**
   - **Pérdida de datos menor** (error de un usuario, borrado accidental):
     - Restaurar desde el backup más reciente el registro específico.
     - Usar consulta SQL directa en Supabase.
   - **Pérdida de datos mayor** (tabla completa dañada, error masivo):
     - Restaurar desde el backup más reciente.
     - Supabase Dashboard → Database → Backups → Restore.
     - Tiempo estimado: 1-4 horas dependiendo del tamaño.
     - Comunicar a usuarios: "Hemos tenido un incidente técnico. Restauramos los datos al estado del [fecha]. Disculpa las molestias."
   - **Pérdida total** (desastre, corrupción general):
     - Restaurar desde el backup semanal manual.
     - Ejecutar script de restauración:
       ```
       psql --dbname=postgresql://xxxx:xxxx@xxxx.supabase.co:5432/postgres < backup-dexpert-YYYY-MM-DD.sql
       ```
     - Tiempo estimado: 2-8 horas.
     - Comunicación extendida a usuarios y stakeholders.

5. **Prueba de recuperación (trimestral)**
   - Cada 3 meses, restaurar el backup más reciente en un ambiente de prueba.
   - Verificar que los datos son íntegros (consultar usuarios, proyectos, transacciones).
   - Documentar resultados y tiempo de recuperación.

### Métricas de Éxito

- Backup automático generado 100% de los días.
- Backup manual semanal sin fallas.
- Prueba de recuperación trimestral exitosa en < 2 horas.
- Datos recuperados sin pérdida en simulacros.

### Notas Importantes

- No almacenar backups en el mismo servidor que la base de datos.
- Las credenciales de BD para backup deben estar en un lugar seguro (password manager corporativo).
- En caso de restauración, los datos posteriores al backup se pierden. Sopesar el impacto.

## SOP-011: Gestión de Créditos PYME

### Objetivo

Administrar correctamente el sistema de créditos de las PYMEs, asegurando que los créditos se otorguen, consuman y expiren según las reglas de negocio.

### Responsable

- **Sistema automático** (asignación y consumo)
- **Soporte** (consultas y resolución de discrepancias)
- **CTO** (verificación del funcionamiento del sistema)

### Herramientas Necesarias

- Panel de administración (ver créditos de PYMEs)
- Base de datos (tabla `pyme_credits`)
- Dashboard de Wompi (para verificar compras)

### Pasos

1. **Asignación automática de créditos**
   - Cuando un PYME compra un plan via Wompi:
     - Webhook procesa la compra.
     - El sistema actualiza o crea un registro en `pyme_credits`:
       - `credits_available += créditos del plan`
       - `expires_at = fecha_actual + 12 meses`
     - Se registra en `credit_purchases`.
   - Planes y créditos:
     - Growth Light ($14.99): 4 créditos
     - Growth ($24.99): 8 créditos
     - Pro ($49.99): 20 créditos
     - Enterprise ($99.99): 50 créditos

2. **Consumo de créditos**
   - Cada vez que un PYME publica un proyecto, el sistema:
     - Verifica que `credits_available > 0`.
     - Si hay créditos: resta 1, incrementa `credits_used`.
     - Si no hay créditos: muestra mensaje "No tienes créditos disponibles. Compra más créditos para publicar."
   - Si el proyecto se cierra sin completarse, el crédito no se reembolsa automáticamente. (El PYME puede solicitar reembolso vía SOP-002).

3. **Verificación de saldo**
   - El PYME puede ver su saldo en el dashboard (créditos disponibles y usados).
   - El admin puede ver el saldo de cualquier PYME en el panel de administración.

4. **Expiración de créditos**
   - Sistema automático: al publicar un proyecto, verificar si los créditos han expirado.
   - Si `expires_at < fecha_actual`:
     - Los créditos disponibles se marcan como expirados (`credits_expired`).
     - El PYME no puede publicar hasta comprar más créditos.
   - Notificar al PYME 30 días antes, 7 días antes, y al expirar.

5. **Resolución de discrepancias**
   - Si un PYME reporta que sus créditos no coinciden:
     - Soporte revisa `credit_purchases` para ver las compras.
     - Revisa `projects` para ver los proyectos publicados.
     - Calcula manualmente el saldo esperado.
     - Si hay error, el admin corrige directamente en `pyme_credits`.

### Métricas de Éxito

- 100% de asignaciones automáticas correctas.
- 0 discrepancias no resueltas mayores a 24 horas.
- PYMEs notificadas antes de la expiración de créditos.

### Notas Importantes

- Los créditos son intransferibles entre PYMEs.
- No hay límite de créditos acumulables (pueden comprar y acumular).
- Al hacer reembolso, descontar los créditos no usados proporcionalmente.

## SOP-012: Proceso de Evaluación de Simulaciones

### Objetivo

Garantizar que las evaluaciones generadas por IA en las simulaciones de estudiantes sean consistentes, justas y útiles para el desarrollo profesional del estudiante.

### Responsable

- **Sistema automático** (evaluación por IA)
- **Admin** (revisión de evaluaciones reportadas)

### Herramientas Necesarias

- Sistema de simulación (AI + evaluación)
- Panel de administración (para revisar evaluaciones)

### Pasos

1. **Inicio de simulación**
   - El estudiante selecciona un escenario de simulación de la lista disponible.
   - El sistema carga el brief del escenario, objetivos, restricciones y skills requeridos.
   - La IA (Groq/Llama-3) genera un mensaje de bienvenida del "cliente virtual".
   - Se crea una sesión en `simulation_sessions` con estado `IN_PROGRESS`.

2. **Durante la simulación**
   - El estudiante conversa con el cliente IA.
   - La IA responde en carácter, usando español salvadoreño coloquial.
   - La IA puede generar change requests dinámicos para simular cambios de alcance reales.
   - El estudiante puede hacer preguntas, negociar, proponer soluciones.
   - Todos los mensajes se guardan en `simulation_messages`.

3. **Finalización y evaluación**
   - El estudiante sube un entregable (URL) y finaliza la simulación.
   - El sistema envía el historial completo a la IA evaluadora (Groq).
   - La IA evalúa en 5 criterios:
     1. **Comprensión del brief** — ¿Entendió los requisitos del cliente?
     2. **Comunicación profesional** — ¿Se comunicó de forma clara y profesional?
     3. **Negociación y manejo de objeciones** — ¿Manejó bien los cambios y objeciones?
     4. **Creatividad y propuestas de solución** — ¿Propuso soluciones creativas?
     5. **Gestión de cambios** — ¿Manejó adecuadamente los change requests?
   - Cada criterio recibe un puntaje de 0-100.
   - Se genera un score total (promedio de los 5 criterios).
   - Se generan: fortalezas (3), áreas de mejora (3), resumen general.
   - Todo se guarda en `evaluations`.

4. **Post-evaluación**
   - El estudiante ve su evaluación completa en la interfaz.
   - Se crea automáticamente una entrada en `portfolio_entries` con `source_type = 'simulation'`.
   - Se actualiza `student_experience`:
     - XP ganada = `(score_total * 0.7) + (mensajes_enviados * 5)`
     - Se verifica si sube de nivel.
   - El estudiante puede publicar la entrada en su portafolio público.

5. **Revisión de evaluaciones (admin)**
   - Si un estudiante reporta que su evaluación fue injusta o incorrecta:
     - Admin revisa el historial de la simulación.
     - Admin puede:
       - Confirmar la evaluación (explicando al estudiante por qué es correcta).
       - Solicitar una re-evaluación (ejecutar evaluación manualmente con diferentes parámetros).
       - Ajustar manualmente el score (solo en casos excepcionales y documentados).

### Métricas de Éxito

- 95%+ de estudiantes completan la simulación sin errores técnicos.
- < 1% de evaluaciones reportadas como injustas.
- Tiempo de respuesta de evaluación < 30 segundos desde que el estudiante finaliza.
- Correlación positiva entre score de simulación y desempeño en proyectos reales (medir trimestralmente).

### Notas Importantes

- La evaluación es automatizada por IA y no es perfecta. Siempre considerar el contexto al revisar reportes.
- Los escenarios de simulación están pre-seleccionados en `seed_scenarios.sql`. Para agregar nuevos, seguir el mismo formato.
- La XP se calcula automáticamente y no se puede modificar manualmente (solo por admin en casos excepcionales).
